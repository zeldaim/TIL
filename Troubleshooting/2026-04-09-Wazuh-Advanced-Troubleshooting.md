# 🛡️ Wazuh SIEM 구축 및 트러블슈팅 종합 리포트

## 1. 실습 개요
- **목표:** Wazuh Manager와 Kali Agent 연동 및 MITRE ATT&CK 기반 탐지 룰 구현
- **환경:** - **Virtualization:** VMware (D 드라이브 기반)
  - **Container:** Docker / Docker-compose
  - **OS:** Rocky Linux 9 (Manager), Kali Linux (Agent)

---

## 2. 트러블슈팅 사례 (Troubleshooting)

### ✅ 이슈 1: 커스텀 탐지 룰(ID: 10001) 미발동
- **현상:** 공격 로그는 정상 유입되나, 커스텀 룰(Level 10)이 무시되고 기본 룰(Level 5)만 발동함.
- **분석:** 부모 룰(`if_matched_sid`)을 `5710`으로 설정했으나, 실제 로그 시그니처는 `5760`과 매칭되어 조건 불일치 발생.
- **해결:** `if_matched_sid`를 **`5760`**으로 변경하여 탐지 우선순위 정상화 및 동작 확인.

### ✅ 이슈 2: 물리 디스크 용량 부족으로 인한 VM 파일 손상
- **현상:** VMware 스냅샷 통합 과정 중 D 드라이브 용량 부족으로 `The specified virtual disk needs repair` 에러 발생.
- **분석:** 임시 공간 부족으로 파일 잠금(`.lck`) 폴더가 해제되지 않고, 가상 디스크 인덱스 정보가 꼬임.
- **조치:** 1. **공간 확보:** 불필요 파일 삭제로 약 8GB 확보.
  2. **잠금 해제:** VM 폴더 내의 `.lck` 폴더 및 파일 수동 삭제.
  3. **디스크 수리:** `vmware-vdiskmanager.exe -R` 실행 후 안정적인 스냅샷 지점으로 `Go To` 복구.

### ✅ 이슈 3: Kali 에이전트 등록 및 통신 장애
- **현상:** Agent 설치 후 서비스는 구동되나 Manager와 'Active' 통신이 이루어지지 않음.
- **분석:** `systemctl status`는 프로세스 실행만 보장할 뿐, 실제 통신 여부는 `ossec.log`의 핸드쉐이크 과정을 확인해야 함.
- **조치:** 1. **연동 재수행:** Manager에서 에이전트 등록 후 추출된 키를 Kali Agent에 재등록.
  2. **네트워크 설정:** Manager IP 오기입 수정 및 포트 `1514` 방화벽 허용 확인.
  3. **ID 충돌 정리:** 기존 등록 정보를 삭제하여 호스트명 기반 자동 등록 시 충돌 방지.

### ✅ 이슈 4: Wazuh와 Kibana 간 포트 충돌 및 Docker 포워딩
- **현상:** 호스트 시스템(Rocky Linux)의 `9200` 포트가 이미 사용 중이라 Elasticsearch 컨테이너 실행 실패.
- **해결:** Docker Compose의 포트 매핑을 `"9201:9200"`으로 변경.
- **핵심 원리:**
  - **호스트 포트(9201):** 외부 브라우저나 서비스가 접근하는 접점.
  - **컨테이너 포트(9200):** 내부 엔진이 실제로 통신하는 포트.
  - **결과:** 호스트의 충돌을 피하면서 컨테이너 내부 통신 구조는 유지함.

## 3. 최종 결론
금일 실습을 통해 **로그 분석(Rule), 자원 관리(Disk), 네트워크 구조(Port Forwarding), 에이전트 아키텍처**에 대한 실무적 이해를 얻음. 특히 문제 발생 시 단순히 결과에 매몰되지 않고 `ossec.log`를 정밀 분석하는 습관이 트러블슈팅의 핵심임을 체득함.
