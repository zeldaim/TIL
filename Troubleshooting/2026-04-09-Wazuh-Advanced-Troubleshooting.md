\# 🛡️ Wazuh SIEM 구축 및 트러블슈팅 종합 리포트



\## 1. 실습 개요

\- \*\*목표:\*\* Wazuh Manager와 Kali Agent 연동 및 MITRE ATT\&CK 기반 탐지 룰 구현

\- \*\*환경:\*\* VMware(D 드라이브), Docker, Rocky Linux 9, Kali Linux



\---



\## 2. 트러블슈팅 사례 (Troubleshooting)



\### 이슈 1: 커스텀 탐지 룰(ID: 10001) 미발동

\- \*\*현상:\*\* 공격 로그는 유입되나 커스텀 룰(Level 10)이 무시되고 기본 룰(Level 5)만 발동함.

\- \*\*분석:\*\* 부모 룰(`if\_matched\_sid`)을 `5710`으로 설정했으나, 실제 시스템 로그는 `5760` 시그니처와 매칭됨.

\- \*\*해결:\*\* `if\_matched\_sid`를 \*\*`5760`\*\*으로 변경하여 정상 탐지 확인.



\### 이슈 2: 물리 디스크 용량 부족으로 인한 VM 파일 손상

\- \*\*현상:\*\* VMware 스냅샷 통합 중 D 드라이브 용량 부족으로 `The specified virtual disk needs repair` 에러 발생.

\- \*\*분석:\*\* 임시 공간 부족으로 인해 파일 잠금(`.lck`) 폴더가 해제되지 않고 디스크 인덱스가 꼬임.

\- \*\*조치:\*\* 1. \*\*물리 공간 확보:\*\* 불필요 파일 삭제로 약 8GB 확보.

&#x20; 2. \*\*잠금 강제 해제:\*\* VM 폴더 내의 `.lck` 폴더 및 파일 삭제.

&#x20; 3. \*\*디스크 수리:\*\* `vmware-vdiskmanager.exe -R` 실행 및 안정적인 이전 스냅샷 지점으로 `Go To` 복구.



\### 이슈 3: Kali 에이전트 등록 및 통신 장애

\- \*\*현상:\*\* Kali Agent 설치 후 Manager와 통신이 이루어지지 않음.

\- \*\*분석:\*\* `systemctl status`는 프로세스 실행만 보여줄 뿐, 실제 통신 여부는 `ossec.log`를 확인해야 함.

\- \*\*조치:\*\*

&#x20; 1. \*\*연동 프로세스 수행:\*\* Manager에서 에이전트 등록 -> 키 추출 -> Kali에서 키 등록 과정 재수행.

&#x20; 2. \*\*네트워크 설정:\*\* Manager IP 설정 오류 수정 및 포트 `1514` 방화벽 허용 확인.

&#x20; 3. \*\*ID 중복 해결:\*\* 에이전트 재설치 시 Manager에 기존 ID 정보가 남아있어 발생하는 충돌을 정리하고 호스트명 기반 자동 등록 확인.



\### 이슈 4: Wazuh와 Kibana 간 포트 충돌 및 Docker 포워딩

\- \*\*현상:\*\* 호스트 시스템(Rocky Linux)의 9200 포트가 이미 점유되어 있어 Kibana/Elasticsearch 연동 실패.

\- \*\*해결:\*\* 포트 포워딩 설정을 `"9201:9200"`으로 변경.

\- \*\*핵심 원리:\*\*

&#x20; - \*\*호스트 포트(9201):\*\* 외부에서 접근하는 관문.

&#x20; - \*\*컨테이너 포트(9200):\*\* Elasticsearch 내부 엔진이 사용하는 고유 포트.

&#x20; - \*\*결과:\*\* 컨테이너 내부 설정은 유지하면서 호스트의 9201로 들어오는 신호를 컨테이너 9200으로 연결하여 충돌 해결.

\## 4. 최종 결론

금일 실습을 통해 \*\*로그 분석(Rule), 자원 관리(Disk), 네트워크 구조(Port Forwarding), 에이전트 아키텍처\*\*에 대한 깊은 이해를 얻음. 특히 에러 발생 시 로그(`ossec.log`)를 분석하는 습관이 문제 해결의 핵심임을 체득함.
